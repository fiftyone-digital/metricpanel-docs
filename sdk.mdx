The `@metricpanel/sdk` package gives your application full control over initialization, consent state, pageview timing, and every event call. Use it when your product requires application-managed consent flows, manual route tracking, or custom event orchestration that goes beyond what the hosted script provides automatically.

## When to use the SDK vs the hosted script

Choose `@metricpanel/sdk` when your application needs to:

- manage consent state programmatically rather than through HTML attributes
- decide exactly when pageviews fire during a client-side route transition
- control bundle size and tree-shaking
- integrate with React Native or other non-browser runtimes

Use the [hosted tracking script](/tracking-script) when you want automatic pageviews, outbound-link capture, and a global `window.metricpanel` API without any build step.

## Package availability

<Warning>
  `@metricpanel/sdk` currently ships as a private workspace package inside the MetricPanel repository. It is **not yet published** to the public npm registry. For an external website, use the hosted tracking script until a versioned public package is announced.
</Warning>

For a self-hosted checkout or another app inside the Bun workspace, declare the dependency and install:

```json package.json
{
  "dependencies": {
    "@metricpanel/sdk": "workspace:*"
  }
}
```

```bash
bun install
```

## Installation and initialization

Create one SDK instance for the entire lifetime of the browser application. Do not create a new instance on every render or route change.

```ts
import { createMetricPanel } from '@metricpanel/sdk'

export const metricpanel = createMetricPanel({
  websiteId: 'YOUR_WEBSITE_ID',
  apiUrl: 'https://api.your-metricpanel-domain.com/api',
  respectDoNotTrack: true,
})
```

`websiteId` is required. `apiUrl` must be the base URL that owns the `/events` ingestion route — the SDK appends `/events` automatically when it sends data.

## Configuration options

| Option | Default | Purpose |
|---|---|---|
| `websiteId` | — | Public website identifier; required |
| `apiUrl` | `/api` | Base URL for event ingestion |
| `debug` | `false` | Log SDK decisions in the browser console |
| `cookieless` | `false` | Use ephemeral visitor and session identifiers instead of persistent cookies |
| `respectDoNotTrack` | `true` | Stop initialization when the browser advertises Do Not Track |
| `anonymizeIP` | `false` | Send the privacy flag that disables or reduces IP-derived processing |
| `waitForConsent` | `false` | Queue all calls until your application explicitly grants consent |
| `trackHashRoutes` | `false` | Include URL hashes when a hash router uses them as navigable route state |

<Note>
  Your legal basis and consent design are application decisions. Read the [Privacy and consent](/privacy-consent) guide before choosing defaults for `waitForConsent`, `cookieless`, and `respectDoNotTrack`.
</Note>

## Track pageviews

Call `pageview()` once after every completed client-side route transition. Pass options when you need to override the values the SDK would otherwise read from the browser:

```ts
// Minimal — reads path, title, and referrer from the browser
await metricpanel.pageview()

// Explicit — useful for SPAs where the DOM hasn't updated yet
await metricpanel.pageview({
  path: '/pricing',
  title: 'Pricing',
  referrer: 'https://example.com/',
})
```

## Track events, goals, and revenue

Use `event()` for product actions, `goal()` for defined conversions, and `revenue()` when money is captured or recognized.

<CodeGroup>
```ts Custom event
await metricpanel.event('signup_completed', {
  plan: 'pro',
  invited_by_team: true,
})
```

```ts Goal
await metricpanel.goal({
  name: 'trial_started',
  value: 4900,
  properties: { plan: 'pro' },
})
```

```ts Revenue
await metricpanel.revenue({
  amount: 4900,
  currency: 'usd',
  productId: 'pro-monthly',
  productName: 'Pro monthly',
})
```
</CodeGroup>

<Tip>
  Revenue amounts and goal values use the smallest currency unit. For USD, `4900` means `$49.00`. Custom events accept at most ten flat properties with `string`, `number`, `boolean`, or `null` values — nested objects and arrays are not supported.
</Tip>

Before committing to production event names, read the [Events](/events), [Goals](/goals), and [Revenue](/revenue) guides.

## Wait for consent

Set `waitForConsent: true` to queue all SDK calls until your application explicitly grants consent. This lets you initialize the SDK at startup without sending any data until the visitor accepts your consent banner.

```ts
const metricpanel = createMetricPanel({
  websiteId: 'YOUR_WEBSITE_ID',
  apiUrl: 'https://api.your-metricpanel-domain.com/api',
  waitForConsent: true,
})

// Queued — nothing is sent yet
await metricpanel.event('pricing_viewed')

// Visitor accepts your consent banner
await metricpanel.grantConsent() // initializes the SDK and flushes the queue

// Visitor withdraws consent later
metricpanel.revokeConsent() // stops tracking and clears SDK identifiers
```

Use `hasConsent()` when your banner UI needs to reflect the current SDK consent state. Revoking consent does not replace account-level visitor deletion or retention workflows.

## Pass attribution into checkout

Call `getStripeMetadata()` to retrieve the current attribution context and attach it to your checkout request before the visitor leaves your page:

```ts
const metadata = metricpanel.getStripeMetadata()

await fetch('/api/create-checkout', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({
    priceId: 'price_123',
    metricpanel: metadata,
  }),
})
```

The metadata object includes the website ID and, when available, visitor ID, session ID, UTM parameters, and supported ad-click identifiers. Store it in your payment provider's metadata field so the webhook can recover attribution after a successful charge. See [Integrations](/integrations) for the full webhook setup.

## React Native and native JavaScript

Use the DOM-free entrypoint for React Native or any environment that does not expose browser globals. Provide a complete ingestion URL and a storage adapter:

```ts
import { MetricPanelNativeSDK } from '@metricpanel/sdk/react-native'

const metricpanel = new MetricPanelNativeSDK({
  websiteId: 'YOUR_WEBSITE_ID',
  apiUrl: 'https://api.your-metricpanel-domain.com/api',
  storage: yourAsyncStorageAdapter,
})

await metricpanel.screen('Upgrade', { path: '/screen/upgrade' })
await metricpanel.event('upgrade_tapped', { plan: 'pro' })
```

The native entrypoint does not read browser globals. It supports manual `screen()`, `pageview()`, `event()`, `goal()`, and `revenue()` calls and can accept an API token for environments where token-authenticated ingestion is required.

## Verification checklist

Before shipping to production, confirm each of the following:

<Steps>
  <Step title="Single instance">
    The SDK is created once and only in a browser or native runtime — not inside a render function or route component.
  </Step>
  <Step title="Matching credentials">
    The website ID and API base both belong to the same MetricPanel website.
  </Step>
  <Step title="Manual pageviews">
    A pageview call fires after every client-side route transition.
  </Step>
  <Step title="Clean properties">
    Event properties are flat and contain no sensitive personal data such as email addresses, passwords, or payment details.
  </Step>
  <Step title="Consent and privacy settings">
    The `waitForConsent`, `cookieless`, and `respectDoNotTrack` values match your published privacy policy.
  </Step>
  <Step title="Revenue attribution">
    Checkout metadata reaches the payment provider before you rely on revenue attribution reports.
  </Step>
</Steps>
