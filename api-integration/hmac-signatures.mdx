HMAC (Hash-based Message Authentication Code) signatures give you a cryptographic guarantee that every event request your server sends to MetricPanel is authentic and hasn't been modified in transit. When you include an HMAC signature, MetricPanel verifies it before processing the event — any request with a missing or invalid signature is rejected with `403 Forbidden`.

## What HMAC Signatures Provide

Signing your ingestion requests ensures:

- **Authenticity** — requests genuinely originate from your authorized server code, not a third party.
- **Integrity** — the payload has not been modified between your server and MetricPanel's API runtime.
- **Security** — your analytics pipeline is protected even if network traffic is intercepted.

<Note>
  Send signed ingestion requests to the API runtime that owns `/api/events` in your deployment. In a split dashboard/API setup, that is your API host, not the dashboard or script host.
</Note>

<Steps>
  <Step title="Open your website settings">
    Navigate to your website in the MetricPanel dashboard.
  </Step>
  <Step title="Go to the API & Security tab">
    Find the **API & Security** tab in the website settings sidebar.
  </Step>
  <Step title="Locate the HMAC Signature Secret section">
    Scroll to the **HMAC Signature Secret** section.
  </Step>
  <Step title="Reveal and copy the secret">
    Click **Reveal Secret** (owners only). Copy the value into your environment variables or secrets manager.
  </Step>
</Steps>

<Warning>
  Never commit your HMAC secret to version control or expose it in client-side code. Always store it in environment variables or a dedicated secrets manager. The secret starts with the prefix `hmac_` — you must strip this prefix before using it to sign requests (see examples below).
</Warning>

## Implementation Examples

Use the language examples below to add HMAC signing to your existing event ingestion code. Every example follows the same three steps: serialize the payload to a consistent JSON string, compute `HMAC-SHA256` over that string with the secret (minus the `hmac_` prefix), and attach the resulting `sha256=<hex>` value in the `X-Signature` header.

<CodeGroup>
  ```javascript JavaScript / Node.js
  import crypto from 'crypto'

  // Load your HMAC secret from environment variables
  const HMAC_SECRET = process.env.METRICPANEL_HMAC_SECRET

  // Build your event payload
  const payload = {
    event: 'pageview',
    url: '/blog/hello-world',
    referrer: 'https://google.com',
    timestamp: Date.now(),
  }

  // Serialize to the exact string you will send in the request body
  const payloadString = JSON.stringify(payload)

  // Strip the 'hmac_' prefix before signing
  const cleanSecret = HMAC_SECRET.replace('hmac_', '')

  // Compute the HMAC-SHA256 signature
  const hmac = crypto.createHmac('sha256', cleanSecret)
  hmac.update(payloadString)
  const signature = 'sha256=' + hmac.digest('hex')

  // Send the request with the signature header
  const response = await fetch('https://api.example.com/api/events', {
    method: 'POST',
    headers: {
      'Content-Type': 'application/json',
      'X-Signature': signature,
      'X-API-Token': process.env.METRICPANEL_API_TOKEN,
    },
    body: payloadString, // Must be the identical string used for signing
  })

  if (!response.ok) {
    console.error('Event tracking failed:', await response.text())
  }
  ```

  ```python Python
  import hmac
  import hashlib
  import json
  import os
  import requests
  from datetime import datetime

  # Load your HMAC secret from environment variables
  HMAC_SECRET = os.getenv('METRICPANEL_HMAC_SECRET')

  # Build your event payload
  payload = {
      'event': 'pageview',
      'url': '/blog/hello-world',
      'referrer': 'https://google.com',
      'timestamp': int(datetime.now().timestamp() * 1000)
  }

  # Serialize with compact separators for consistent output
  payload_string = json.dumps(payload, separators=(',', ':'))

  # Strip the 'hmac_' prefix before signing
  clean_secret = HMAC_SECRET.replace('hmac_', '', 1)

  # Compute the HMAC-SHA256 signature
  signature_bytes = hmac.new(
      clean_secret.encode('utf-8'),
      payload_string.encode('utf-8'),
      hashlib.sha256
  ).hexdigest()

  signature = f'sha256={signature_bytes}'

  # Send the request with the signature header
  response = requests.post(
      'https://api.example.com/api/events',
      headers={
          'Content-Type': 'application/json',
          'X-Signature': signature,
          'X-API-Token': os.getenv('METRICPANEL_API_TOKEN')
      },
      data=payload_string  # Must be the identical string used for signing
  )

  if not response.ok:
      print(f'Event tracking failed: {response.text}')
  ```

  ```php PHP
  <?php

  // Load your HMAC secret from environment variables
  $hmacSecret = getenv('METRICPANEL_HMAC_SECRET');

  // Build your event payload
  $payload = [
      'event' => 'pageview',
      'url' => '/blog/hello-world',
      'referrer' => 'https://google.com',
      'timestamp' => round(microtime(true) * 1000)
  ];

  // Serialize to the exact string you will send in the request body
  $payloadString = json_encode($payload, JSON_UNESCAPED_SLASHES);

  // Strip the 'hmac_' prefix before signing
  $cleanSecret = str_replace('hmac_', '', $hmacSecret);

  // Compute the HMAC-SHA256 signature
  $signatureHash = hash_hmac('sha256', $payloadString, $cleanSecret);
  $signature = 'sha256=' . $signatureHash;

  // Send the request with the signature header
  $ch = curl_init('https://api.example.com/api/events');
  curl_setopt_array($ch, [
      CURLOPT_POST => true,
      CURLOPT_RETURNTRANSFER => true,
      CURLOPT_HTTPHEADER => [
          'Content-Type: application/json',
          'X-Signature: ' . $signature,
          'X-API-Token: ' . getenv('METRICPANEL_API_TOKEN')
      ],
      CURLOPT_POSTFIELDS => $payloadString  // Must be the identical string used for signing
  ]);

  $response = curl_exec($ch);
  $httpCode = curl_getinfo($ch, CURLINFO_HTTP_CODE);
  curl_close($ch);

  if ($httpCode !== 200) {
      error_log('Event tracking failed: ' . $response);
  }
  ```

  ```ruby Ruby
  require 'openssl'
  require 'json'
  require 'net/http'
  require 'uri'

  # Load your HMAC secret from environment variables
  hmac_secret = ENV['METRICPANEL_HMAC_SECRET']

  # Build your event payload
  payload = {
    event: 'pageview',
    url: '/blog/hello-world',
    referrer: 'https://google.com',
    timestamp: (Time.now.to_f * 1000).to_i
  }

  # Serialize to the exact string you will send in the request body
  payload_string = JSON.generate(payload)

  # Strip the 'hmac_' prefix before signing
  clean_secret = hmac_secret.sub(/^hmac_/, '')

  # Compute the HMAC-SHA256 signature
  digest = OpenSSL::Digest.new('sha256')
  signature_hash = OpenSSL::HMAC.hexdigest(digest, clean_secret, payload_string)
  signature = "sha256=#{signature_hash}"

  # Send the request with the signature header
  uri = URI('https://api.example.com/api/events')
  http = Net::HTTP.new(uri.host, uri.port)
  http.use_ssl = true

  request = Net::HTTP::Post.new(uri.path)
  request['Content-Type'] = 'application/json'
  request['X-Signature'] = signature
  request['X-API-Token'] = ENV['METRICPANEL_API_TOKEN']
  request.body = payload_string  # Must be the identical string used for signing

  response = http.request(request)

  unless response.is_a?(Net::HTTPSuccess)
    puts "Event tracking failed: #{response.body}"
  end
  ```

  ```go Go
  package main

  import (
      "bytes"
      "crypto/hmac"
      "crypto/sha256"
      "encoding/hex"
      "encoding/json"
      "fmt"
      "io"
      "net/http"
      "os"
      "strings"
      "time"
  )

  type EventPayload struct {
      Event     string `json:"event"`
      URL       string `json:"url"`
      Referrer  string `json:"referrer"`
      Timestamp int64  `json:"timestamp"`
  }

  func generateSignature(payload []byte, secret string) string {
      // Strip the 'hmac_' prefix before signing
      cleanSecret := strings.TrimPrefix(secret, "hmac_")

      // Compute the HMAC-SHA256 signature
      h := hmac.New(sha256.New, []byte(cleanSecret))
      h.Write(payload)
      return "sha256=" + hex.EncodeToString(h.Sum(nil))
  }

  func main() {
      // Load your HMAC secret from environment variables
      hmacSecret := os.Getenv("METRICPANEL_HMAC_SECRET")

      // Build your event payload
      payload := EventPayload{
          Event:     "pageview",
          URL:       "/blog/hello-world",
          Referrer:  "https://google.com",
          Timestamp: time.Now().UnixMilli(),
      }

      // Serialize to JSON — use the same bytes for signing and the request body
      payloadJSON, err := json.Marshal(payload)
      if err != nil {
          fmt.Printf("Error marshaling payload: %v\n", err)
          return
      }

      signature := generateSignature(payloadJSON, hmacSecret)

      // Build and send the request
      req, err := http.NewRequest(
          "POST",
          "https://api.example.com/api/events",
          bytes.NewBuffer(payloadJSON),
      )
      if err != nil {
          fmt.Printf("Error creating request: %v\n", err)
          return
      }

      req.Header.Set("Content-Type", "application/json")
      req.Header.Set("X-Signature", signature)
      req.Header.Set("X-API-Token", os.Getenv("METRICPANEL_API_TOKEN"))

      client := &http.Client{}
      resp, err := client.Do(req)
      if err != nil {
          fmt.Printf("Error sending request: %v\n", err)
          return
      }
      defer resp.Body.Close()

      if resp.StatusCode != http.StatusOK {
          body, _ := io.ReadAll(resp.Body)
          fmt.Printf("Event tracking failed: %s\n", string(body))
      }
  }
  ```
</CodeGroup>

## Required Request Headers

Every signed request must include these three headers:

| Header | Required | Description |
|---|---|---|
| `Content-Type` | Yes | Must be `application/json` |
| `X-Signature` | Yes | Your HMAC signature in the format `sha256=<hex>` |
| `X-API-Token` | Yes | Your website API token for authentication |

---

## Signature Format

The `X-Signature` header value must follow this exact format:

```text
sha256=<hex_encoded_signature>
```

**Example:**

```text
sha256=3f8d0e9c4b2a1f7e6d5c4b3a2f1e0d9c8b7a6f5e4d3c2b1a0f9e8d7c6b5a4f3
```

## Important Notes

### Payload Consistency

The signature is computed from the **exact byte sequence** of the JSON string you send as the request body. If anything changes between the string you signed and the body that arrives at the server, verification will fail.

<Warning>
  Always use the **same string variable** for both the signature calculation and the `body` of your HTTP request. Avoid re-serializing the payload between those two steps — even a single added space or reordered key will produce a different signature.
</Warning>

Keep these consistency rules in mind:

- Use compact JSON serialization with no extra whitespace (most language defaults do this).
- Keep key order identical — don't sort or transform the object between signing and sending.
- Use UTF-8 encoding end-to-end.

### Secret Handling

- **Remove the `hmac_` prefix** from the raw secret before you pass it to your HMAC function. All examples above demonstrate this.
- Never log the full secret value — log only the first several characters if you need to debug.
- Store the secret in environment variables or a dedicated secrets manager, never hardcoded in source files.
- Rotate your secret regularly (see Security Best Practices below).

### Grace Period During Rotation

When you rotate your HMAC secret in the dashboard, MetricPanel keeps both the old and the new secret valid for a **24-hour grace period**. This means:

- Requests signed with the old secret continue to be accepted during the grace period, giving you time to update your services without downtime.
- Once the grace period expires, the old secret stops working.
- If MetricPanel accepts your request but returns a warning that the old secret was used, update all of your services to the new secret before the 24 hours elapse.

## Testing Your Implementation

Work through these three scenarios before deploying to production.

<Steps>
  <Step title="Test with a valid signature">
    Set your credentials and run your implementation:

    ```bash
    export METRICPANEL_HMAC_SECRET="hmac_your_secret_here"
    export METRICPANEL_API_TOKEN="mtk_your_token_here"

    node send-event.js   # or: python send-event.py, php send-event.php, etc.
    ```

    You should receive a `200 OK` response.
  </Step>
  <Step title="Test with an invalid signature">
    Try sending a request with each of these deliberate errors:

    - A wrong or misspelled secret
    - The `sha256=` prefix missing from the `X-Signature` value
    - The payload modified after signing (add a space, change a value)

    Each case should return `403 Forbidden` with the message `"Invalid signature"`.
  </Step>
  <Step title="Test secret rotation">
    1. Rotate your HMAC secret in the dashboard (**API & Security** tab).
    2. Send a request using the **old** secret — it should still be accepted during the grace period.
    3. Update your integration to the **new** secret and verify requests succeed.
    4. After 24 hours, confirm the old secret is rejected.
  </Step>
</Steps>

## Troubleshooting

<Accordion title="403 Forbidden — Invalid Signature">
  **Possible causes:**

  1. **Wrong secret** — verify you are using the correct secret from the **API & Security** tab.
  2. **Prefix not removed** — confirm you strip `hmac_` from the secret before signing.
  3. **Payload mismatch** — the string used for signing differs from the request body bytes.
  4. **Encoding issues** — use UTF-8 consistently across serialization and the HTTP request.

  **Debug steps:**

  ```javascript
  // Log values to compare against what MetricPanel receives
  console.log('Secret (first 10 chars):', HMAC_SECRET.substring(0, 10))
  console.log('Payload string:', payloadString)
  console.log('Generated signature:', signature)

  // Confirm the body and signed string are identical
  const requestBody = payloadString
  console.log('Bodies match:', requestBody === payloadString)
  ```
</Accordion>

<Accordion title="Signature Accepted with Old Secret Warning">
  You are using the old secret after rotating. The 24-hour grace period is still active, so your requests succeed — but you **must** update your integration to the new secret before the grace period expires. Check your deployment pipeline and update every service that signs requests.
</Accordion>

<Accordion title="401 Unauthorized">
  A `401` response indicates an API token problem, not a signature problem. Check:

  - The `X-API-Token` header is present and contains the correct token value.
  - The token has not been revoked from the **API & Security** tab.
  - For revenue ingestion (`POST /api/v1/revenue/events`), the token has the `ingest` scope.

  Fix the token issue first, then re-test your signature.
</Accordion>

## Security Best Practices

### Store Secrets in Environment Variables

Never hardcode secrets in source files. Use a `.env` file (excluded from version control) or a cloud secrets manager:

```bash
# .env.local — add this file to .gitignore
METRICPANEL_HMAC_SECRET=hmac_abc123...
METRICPANEL_API_TOKEN=mtk_xyz789...
```

### Follow a Secret Rotation Schedule

Rotate your HMAC secret:

- Every **90–180 days** as routine maintenance.
- **Immediately** if you suspect the secret has been compromised.
- When **offboarding** any team member who had access to the secret.
- After **security incidents or audits**.

### Monitor for Anomalies

Set up alerts for:

- Elevated rates of signature verification failures (`403` responses).
- Unusual patterns in failed ingestion requests.
- Any use of an old secret after rotation — this indicates a service that was not updated.

### Enforce Network Security

- Always use HTTPS. Never send signed requests over unencrypted HTTP.
- Never pass secrets in query parameters or request URLs.
- Use encrypted storage for secrets at rest.

## Rate Limiting

Signed requests are subject to the same rate limits as all other requests to `/api/events`:

| Limit | Value |
|---|---|
| Requests per second | 100 per website |
| Requests per hour | 10,000 per website |

Exceeding either limit returns `429 Too Many Requests`. Implement exponential backoff and retry logic in your ingestion code to handle bursts gracefully.
